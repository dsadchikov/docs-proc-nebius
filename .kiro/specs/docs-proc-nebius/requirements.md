# Requirements Document

## Introduction

This specification defines the requirements for **docs-proc-nebius**: a standalone document recognition pipeline deployed on Nebius Serverless AI infrastructure.

**docs-proc-nebius is a standalone product.** Any client can use it directly via HTTP — no dependency on AWS, DynamoDB, or any specific backend framework. The only required external service is Nebius Object Storage (for blueprint persistence and, optionally, document transport).

The lity-backend integration (AWS Lambda → DynamoDB) is an optional client of the standalone API and is described separately in Requirement 7.

The pipeline consists of two components:

1. A **Nebius Serverless AI Endpoint** — a persistent HTTP inference service running Qwen2.5-VL-7B-Instruct via vLLM, fronted by a FastAPI layer (CPU logic) and nginx (reverse proxy). It accepts a document (`base64` inline or `nebius_object` key in NOS) and returns structured JSON extraction results aligned to a blueprint schema.
2. A **Nebius Serverless AI Job** — a one-off containerized batch processing job that reads document IDs from a manifest in Nebius Object Storage, calls the Endpoint for each document, and writes results back to Nebius Object Storage.

This spec is submitted as part of the Nebius Serverless AI Builders Challenge (deadline June 30, 2026).

## Glossary

- **Blueprint**: A versioned JSON document defining extraction rules for a specific document type. Stored in Nebius Object Storage in rich format (`sections` + `inferenceType`); normalized to a flat `fields[]` list by BlueprintStore at load time.
- **Blueprint Catalog** (`_catalog.json`): A single JSON index file at `blueprints/_catalog.json` listing all blueprints with their `status`, `latest_version`, and S3 path. The BlueprintStore reads this file at startup to discover which blueprints to load.
- **Blueprint_Selector**: The system component responsible for matching an input document to an appropriate Blueprint (used in `mode=auto`).
- **BlueprintStore**: The in-memory cache of blueprints, populated from Nebius Object Storage at startup via the Blueprint Catalog, and kept in sync via the CRUD API.
- **Default Blueprint**: A catch-all blueprint with `id: "default"` that attempts to extract all visible fields from any unknown document type. Used as fallback when `mode=auto` finds no matching blueprint.
- **Confidence_Score**: A whole-number integer in the range [0, 100] representing the model's certainty for a classification decision or an extracted field value.
- **Document_Part**: A physically distinct side or page group of a document (e.g., `single`, `page_1_of_3`).
- **Endpoint**: The Nebius Serverless AI Endpoint running the Qwen2.5-VL-7B-Instruct model via vLLM, with FastAPI + nginx in front.
- **Extractor**: The system component within the Endpoint that produces a structured JSON result from a document image.
- **Inference_Job**: The Nebius Serverless AI Job container that performs batch re-recognition.
- **lity-backend**: The existing Lity AWS Lambda backend (Node.js 22.x, eu-north-1) that integrates with the Endpoint via HTTP.
- **Guided_Decoding**: vLLM structured-output mode (`guided_json`) that constrains token generation to a JSON Schema derived from the active Blueprint, guaranteeing schema-valid output without post-hoc parsing.
- **Logprob_Confidence**: A per-field Confidence_Score computed from the token log-probabilities of the generated field value (vLLM `logprobs: true`), replacing model-self-reported confidence.
- **Manifest**: A JSON file in Nebius Object Storage listing the document IDs to be processed by the Inference_Job.
- **MIDV-2020**: A publicly available academic dataset of 1000 synthetic (dummy-data) identity documents across 10 document types, with per-field ground-truth annotations. Used as the contest-compliant evaluation and demo dataset — contains no real personal data.
- **Document_Packet**: A single uploaded file (multi-page PDF) containing one or more logical documents. Packet processing classifies each page, groups consecutive pages into logical documents, and extracts each independently.
- **Evaluation_Job**: A Nebius Serverless AI Job that runs the recognition pipeline over a labeled dataset (MIDV-2020) and produces per-field and per-document-type accuracy metrics.
- **NOS** (Nebius Object Storage): The S3-compatible object storage service on Nebius. Endpoint: `https://storage.eu-north1.nebius.cloud`. Bucket: `your-nos-bucket`.
- **NOS_Presigned_PUT_URL**: A time-limited Nebius Object Storage pre-authenticated URL that grants write access to a specific object key. Generated by lity-backend to upload documents to NOS before recognition.
- **Recognition_Result**: A structured JSON object produced by the Extractor, containing all extracted fields with their values and Confidence_Scores, plus a document-level `document_confidence` and `routing` field.
- **lity-poc-docs**: The AWS S3 bucket in eu-north-1 where lity-backend stores uploaded user documents.
- **Presigned_URL**: A time-limited AWS S3 pre-authenticated URL that grants read access to a specific document object.
- **Rich Blueprint Format**: The on-disk JSON format for blueprints, using `sections` grouping and per-field `inferenceType`/`instruction` metadata, compatible with the `ltu-prp-scheme-front.json` structure.
- **VLM**: Vision-Language Model. In this spec, refers to Qwen2.5-VL-7B-Instruct, the model deployed on the Endpoint.

## Requirements

### Requirement 1: Nebius Serverless AI Endpoint Deployment

**User Story:** As a developer, I want to deploy a persistent HTTP inference endpoint on Nebius Serverless AI, so that any client can perform document recognition via HTTP without managing GPU infrastructure or depending on any specific cloud provider.

#### Acceptance Criteria

1. THE Endpoint SHALL expose an HTTP POST route at `/recognize` that accepts a JSON request body with the following fields:
   - `document` (required): `{ "type": "base64" | "nebius_object", "value": "<string>", "mime_type": "<string>", "page": <int, optional> }`
   - `mode` (optional, default `"blueprint"`): `"blueprint"` | `"raw"` | `"auto"` | `"double_check"`
   - `blueprint_id` (required when `mode="blueprint"` or `mode="double_check"`): string
   - `options` (optional): `{ "include_confidence": bool, "confidence_mode": "document"|"fields"|"both", "document_type_hint": string, "include_bounding_boxes": bool }`

2. THE Endpoint SHALL expose `GET /inbound/presign?filename=<name>` that returns a NOS presigned PUT URL and NOS object key for direct client upload. This enables standalone clients to upload large documents (including PDFs) without encoding them as base64. IF NOS credentials are not configured, THE Endpoint SHALL return HTTP 503.

3. WHEN `mode="blueprint"` and a valid blueprint is found, THE Endpoint SHALL respond with a Recognition_Result JSON and HTTP 200, containing: `request_id`, `mode`, `blueprint_id`, `document_confidence` (integer, 0–100), `routing` (string), `document_part` (string), `fields` (object with per-field `value` and `confidence`).

4. WHEN `mode="raw"`, THE Endpoint SHALL respond with HTTP 200 containing `mode`, `document_part`, and `raw_text`.

5. WHEN `mode="auto"`, THE Endpoint SHALL classify the document type, look up a matching blueprint, and — if found — return a full blueprint extraction result with an additional `classification` field. If no matching blueprint is found, THE Endpoint SHALL return HTTP 200 with `routing: "escalate_to_operator"` and `raw_text`.

6. WHEN a valid `/recognize` request exceeds 30 seconds, THE Endpoint SHALL return HTTP 504.

7. THE Endpoint SHALL have zero runtime dependencies on AWS services (no S3, DynamoDB, SSM, or any AWS SDK calls). All external I/O is via Nebius Object Storage only.

8. THE Endpoint SHALL run the Qwen2.5-VL-7B-Instruct model via vLLM with: `--dtype bfloat16 --max-model-len 32768 --gpu-memory-utilization 0.85 --limit-mm-per-prompt image=1,video=0`.

9. THE Endpoint SHALL be deployed on Nebius GPU H100 SXM (`--preset 1gpu-16vcpu-200gb`) with `--shm-size 16Gi --disk-size 250Gi --public --auth token`.

10. IF `mode="blueprint"` and `blueprint_id` is missing or not found, THE Endpoint SHALL return HTTP 422.

11. THE Endpoint SHALL remain accessible at a stable URL for the duration of the contest period. The public IP and auth token change on every `nebius ai endpoint create` — update SSM after each redeploy.

### Requirement 2: Blueprint-Based Field Extraction

**User Story:** As a developer, I want the Endpoint to extract structured fields from a document image using a Blueprint schema, so that lity-backend can store structured recognition results in DynamoDB without parsing raw model output.

#### Acceptance Criteria

1. WHEN a document image and a Blueprint are provided to the Extractor, THE Extractor SHALL return a Recognition_Result containing only the fields defined in that Blueprint's `fields` list, and SHALL NOT include fields not defined in the Blueprint.

2. IF a required Blueprint field is absent from the document image, THEN THE Extractor SHALL return that field with `value: null` and `confidence: 0` rather than omitting the field from the Recognition_Result.

3. THE Extractor SHALL assign a Confidence_Score (integer, 0–100) to each extracted field. IF the model returns a value outside [0, 100], THE Extractor SHALL clamp it to the nearest bound. IF the model returns a non-integer, THE Extractor SHALL round to the nearest integer.

4. THE Extractor SHALL assign a document-level `document_confidence` (integer, 0–100) computed as the mean of all field confidence scores, clamped to [0, 100].

5. WHEN a Blueprint field has `instruction` containing a canonical format (e.g., `"YYYY-MM-DD"`, `"M or F"`, `"uppercase as printed"`), the VLM system prompt SHALL include that instruction verbatim per field.

6. IF the model response cannot be parsed as valid JSON (only possible when Guided_Decoding is unavailable — see Requirement 14), THE Extractor SHALL fall back to regex extraction of a JSON object from the raw response before returning `value: null, confidence: 0` for unparseable fields. Per-field confidence SHALL be computed via Logprob_Confidence (Requirement 13); the legacy flat default of 80 SHALL be used only when logprobs are unavailable in the vLLM response.

7. WHEN `options.include_bounding_boxes=true`, THE Extractor SHALL include a `bounding_box: { x, y, width, height }` (normalized 0–1) in each FieldResult where the model provides it.

8. WHEN `mode="double_check"`, THE Extractor SHALL perform a second VLM pass to verify each extracted field and adjust confidence scores accordingly (confirmed fields +10, unconfirmed fields → 0 or corrected value at 60).

### Requirement 3: Confidence Threshold Routing

**User Story:** As a developer building lity-backend integration, I want the Recognition_Result to include a routing value, so that lity-backend can route the result to auto-classification, review, or operator escalation without re-querying the Endpoint.

#### Acceptance Criteria

1. IF `document_confidence` ∈ [85, 100], THEN `routing` SHALL be `"auto_classified"`.
2. IF `document_confidence` ∈ [50, 84], THEN `routing` SHALL be `"review_required"`.
3. IF `document_confidence` ∈ [0, 49], THEN `routing` SHALL be `"escalate_to_operator"`.
4. THE `document_confidence` field SHALL be a whole-number integer in [0, 100] after clamping and rounding.
5. THE `routing` field SHALL always contain exactly one value; the three bands [0,49], [50,84], [85,100] are exhaustive and mutually exclusive.
6. WHEN `options.confidence_mode="document"`, THE Recognition_Result SHALL include `document_confidence` and `routing` but SHALL omit per-field confidence values. WHEN `options.confidence_mode="fields"`, per-field confidences SHALL be included but `document_confidence` and `routing` SHALL be null. WHEN `options.confidence_mode="both"` (default), all confidence values SHALL be included.

### Requirement 4: Document Type Scope

**User Story:** As a lawyer using Lity, I want the pipeline to recognize the document types relevant to the contest submission, so that client documents in case-c124 and residence permit intake workflows can be processed automatically.

#### Acceptance Criteria

1. THE Endpoint SHALL support extraction for the following built-in document types, each identified by `blueprint_id` and stored in NOS: `default`, `passport`, `residence_permit_ltu_front`. These blueprints SHALL be loaded into the BlueprintStore at startup via the Blueprint Catalog.

2. THE `default` blueprint SHALL be a catch-all that attempts to extract all visible fields from any document type. It SHALL be used automatically by `mode=auto` when no other blueprint matches the classified document type.

3. WHEN a Passport document is provided with `blueprint_id: "passport"`, THE Extractor SHALL extract at minimum: `document_type`, `document_number`, `issuing_country`, `surname`, `given_names`, `sex`, `nationality`, `date_of_birth` (YYYY-MM-DD), `date_of_expiry` (YYYY-MM-DD), `mrz_line_1`, `mrz_line_2`.

4. WHEN a Lithuanian Residence Permit (front side) is provided with `blueprint_id: "residence_permit_ltu_front"`, THE Extractor SHALL extract all fields defined across sections `DOCUMENT_METADATA`, `PERSONAL_INFO`, `PERMIT_DETAILS`, and `ADDITIONAL_INFO` in the rich blueprint format, including: `document_type`, `document_part`, `document_number`, `issuing_country`, `document_language`, `surname`, `given_name`, `sex`, `nationality`, `date_of_birth`, `permit_type`, `expiry_date`, `remarks`, `personal_number`, `card_access_number`.

5. IF `blueprint_id` is not found in the BlueprintStore, THE Endpoint SHALL return HTTP 422 per Requirement 1, criterion 11.

6. THE built-in blueprints SHALL be pre-loaded into NOS during initial setup and SHALL NOT require manual API calls to activate.

### Requirement 5: Blueprint Storage Architecture (NOS)

**User Story:** As a developer, I want blueprints and all pipeline artifacts stored in a well-structured Nebius Object Storage bucket, so that blueprints are versioned, discoverable, and the full pipeline has a consistent audit trail.

#### Acceptance Criteria

1. THE system SHALL use a single NOS bucket (`your-nos-bucket`) with the following top-level directory structure:
   ```
   your-nos-bucket/
   ├── blueprints/          ← blueprint definitions and catalog
   ├── inbound/             ← documents uploaded by lity-backend for recognition
   ├── outbound/            ← Recognition_Result JSON files written after recognition
   └── logs/                ← per-request and per-job structured log entries
   ```

2. THE `blueprints/` directory SHALL contain a Blueprint Catalog file at `blueprints/_catalog.json` listing all known blueprints with fields: `id`, `name`, `status` (`active` | `draft` | `deprecated`), `latest_version` (integer), `path` (NOS key to the latest version file), `created_at`, `updated_at`.

3. EACH blueprint SHALL be stored in its own subdirectory: `blueprints/<id>/v<N>.json` where N is the version number. WHEN a blueprint is updated, the previous version file SHALL be preserved and a new version file SHALL be created.

4. THE Rich Blueprint Format for all blueprint files SHALL follow this structure:
   ```json
   {
     "$schema": "http://json-schema.org/draft-07/schema#",
     "id": "<blueprint_id>",
     "name": "<human-readable name>",
     "version": <integer>,
     "status": "active" | "draft" | "deprecated",
     "description": "<string>",
     "extraction_prompt": "<string>",
     "document_parts": ["single"] | ["Front", "Back"] | ["page_1", "page_2"],
     "sections": {
       "<SECTION_NAME>": {
         "<field_name>": {
           "inferenceType": "explicit" | "inferred",
           "instruction": "<canonical format instruction>",
           "required": true | false
         }
       }
     },
     "created_at": "<ISO 8601>",
     "updated_at": "<ISO 8601>"
   }
   ```

5. THE `inbound/` directory SHALL use a time-partitioned path structure: `inbound/YYYY/MM/DD/HH/mm/<document_id>.<ext>`. Documents SHALL be uploaded by lity-backend via a NOS_Presigned_PUT_URL before calling `/recognize`.

6. THE `outbound/` directory SHALL use the same time-partitioned structure: `outbound/YYYY/MM/DD/HH/mm/<document_id>.json`. THE Endpoint SHALL write the Recognition_Result JSON to this path after successful recognition.

7. THE `logs/` directory SHALL use the same time-partitioned structure: `logs/YYYY/MM/DD/HH/mm/`. TWO log entry types SHALL be written:
   - `req_<request_id>.json` — one per `/recognize` call, containing: `request_id`, `timestamp`, `blueprint_id`, `mode`, `document_key`, `document_confidence`, `routing`, `latency_ms`, `model`, `endpoint_version`
   - `job_<job_id>.json` — one per Inference Job run, containing the job summary

8. WHEN NOS credentials are not configured, THE system SHALL operate in local-only mode: no reads or writes to any NOS path, logs are written only to stdout.

### Requirement 6: Blueprint Management API

**User Story:** As a developer, I want a CRUD API for blueprints so that I can add, update, and delete document schemas at runtime without rebuilding the container image.

#### Acceptance Criteria

1. THE Endpoint SHALL expose the following blueprint management routes, all requiring Bearer token authentication:
   - `GET /blueprints` — list all blueprints (returns array of `BlueprintMeta` from in-memory cache)
   - `GET /blueprints/{id}` — get full rich blueprint by ID (404 if not found)
   - `POST /blueprints` — create blueprint in rich format (409 if ID already exists); persists to NOS and updates `_catalog.json`
   - `PUT /blueprints/{id}` — update blueprint (404 if not found); increments version, writes new version file, preserves old, updates `_catalog.json`
   - `DELETE /blueprints/{id}` — marks blueprint as `deprecated` in `_catalog.json`, HTTP 204 (404 if not found); does NOT delete NOS files
   - `POST /blueprints/reload` — reloads all `active` blueprints from NOS into in-memory cache

2. WHEN the Endpoint starts, THE BlueprintStore SHALL read `blueprints/_catalog.json`, then load only blueprints with `status: "active"` by fetching each file at its catalog `path`. IF the catalog is missing, THE BlueprintStore SHALL fall back to scanning `blueprints/*/` for the highest-versioned file.

3. WHEN BlueprintStore loads a rich-format blueprint, it SHALL normalize the `sections` hierarchy into a flat `fields[]` list for use by the Extractor. The normalization SHALL preserve `name`, `inferenceType`, `instruction`, and `required` for each field.

4. THE `BlueprintMeta` response SHALL include: `id`, `name`, `version`, `status`, `description`, `fields_count`, `created_at`, `updated_at`.

5. THE NOS connection SHALL use `signature_version=s3v4`, endpoint `https://storage.eu-north1.nebius.cloud`, region `eu-north1`. Credentials via `S3_ACCESS_KEY` / `S3_SECRET_KEY` environment variables.

### Requirement 7: lity-backend Integration (optional)

**User Story:** As a developer of lity-backend, I want to integrate the Endpoint into the existing AWS Lambda workflow, so that document recognition results are stored in DynamoDB alongside case data.

**This requirement is optional and does not affect the standalone operation of docs-proc-nebius.** The Endpoint has no knowledge of this integration.

#### Acceptance Criteria

1. WHEN lity-backend receives a document recognition trigger, it SHALL:
   a. Fetch the document from `lity-poc-docs` (AWS S3) via `GetObject`
   b. Upload the document to NOS `inbound/YYYY/MM/DD/HH/mm/<documentId>.<ext>` via presigned PUT URL
   c. Call `/recognize` with `document.type: "nebius_object"`, `document.value: <nos_key>`, and `blueprint_id`
   d. Fallback: if NOS upload fails → call `/recognize` with `document.type: "base64"`

2. WHEN THE Endpoint returns HTTP 200, THE lity-backend SHALL store the Recognition_Result in DynamoDB `lity-cases-poc` under `DOC#<documentId>` with: `recognitionResult`, `recognitionRouting`, `recognitionStatus: "completed"`, `recognitionAt`.

3. ALL failures (NOS upload, Endpoint call, DynamoDB write) SHALL be logged as structured JSON and returned as structured error objects — no throw, no retry.

4. THE SSM parameters used by `nebius-adapter.js`:
   - `/lity/nebius-endpoint-url` — Endpoint base URL
   - `/lity/nebius-endpoint-token` — Bearer token (SecureString)
   - `/lity/nebius-s3-access-key` — NOS static key ID (SecureString)
   - `/lity/nebius-s3-secret-key` — NOS static secret key (SecureString)
   - `/lity/nebius-s3-bucket` — NOS bucket name
   - `/lity/docs-bucket-name` — AWS S3 bucket (shared with documents.js)

5. THE `nebius-adapter.js` module SHALL use Node.js 22 built-in `fetch` and `@aws-sdk/client-s3` only — no Nebius SDK. It SHALL be the only place in lity-backend referencing Nebius internals.

### Requirement 8: Nebius Serverless AI Inference Job

**User Story:** As a developer, I want a one-off batch job that re-processes a list of documents and writes results to NOS outbound, so that I can re-recognize documents in bulk without triggering individual HTTP calls from lity-backend.

#### Acceptance Criteria

1. THE Inference_Job SHALL read a Manifest JSON file from `$MANIFEST_PATH` in NOS at startup.
2. IF the Manifest is unreadable, THE Job SHALL log to stdout and exit with code 1.
3. WHEN the Manifest contains N document IDs (N ≥ 1), THE Job SHALL produce exactly N output files in `$OUTPUT_PATH`, named `<document_id>.json`.
4. THE Job SHALL call `/recognize` for each document in manifest order with `document.type: "nebius_object"` pointing to the document's key in `inbound/`.
5. IF the Endpoint call fails after up to `$MAX_RETRIES` (default: 10) attempts, THE Job SHALL write an error record JSON and continue to the next document.
6. WHEN all documents are processed, THE Job SHALL exit with code 0 and log a summary to stdout AND write `logs/YYYY/MM/DD/HH/mm/job_<job_id>.json` to NOS.
7. WHEN the Manifest is empty, THE Job SHALL exit with code 0 without calling the Endpoint.
8. THE Job SHALL be packaged as a Docker container (`nebius-job/Dockerfile`).

### Requirement 9: Document Upload via NOS (Endpoint side)

**User Story:** As a developer, I want the Endpoint to read documents from NOS directly, so that large payloads are not transmitted inline and the 403 problem with public AWS S3 access is eliminated.

#### Acceptance Criteria

1. THE Endpoint SHALL support `document.type: "nebius_object"` — `document.value` is the NOS object key, fetched via boto3 S3 client before processing.

2. THE Endpoint SHALL support PDF via all three document types. WHEN `mime_type: "application/pdf"`, THE Endpoint SHALL convert `document.page` (default: 1) to JPEG using `poppler-utils` before VLM inference.

3. WHEN NOS credentials are not configured (`S3_ACCESS_KEY` empty), `document.type: "nebius_object"` SHALL return HTTP 503 with `{ "message": "Object storage not configured" }`.

4. AFTER successful recognition, THE Endpoint SHALL write the Recognition_Result JSON to `outbound/YYYY/MM/DD/HH/mm/<document_id>.json` in NOS (best-effort, non-blocking — failure SHALL be logged but SHALL NOT affect the HTTP response).

5. AFTER each recognition request, THE Endpoint SHALL write a structured log entry to `logs/YYYY/MM/DD/HH/mm/req_<request_id>.json` in NOS (best-effort, non-blocking).

### Requirement 10: Blueprint Generation from Sample Document

**User Story:** As a developer, I want to generate a draft blueprint by submitting a sample document image, so that I can onboard new document types without writing the JSON schema by hand.

#### Acceptance Criteria

1. THE Endpoint SHALL expose `POST /blueprints/generate` accepting: `document` (same shape as `/recognize`), `blueprint_id` (proposed ID), `name` (human-readable), `description` (optional).

2. WHEN `/blueprints/generate` is called, THE Endpoint SHALL perform a `mode=raw` VLM pass on the document, then a second VLM pass to structure the observed fields into the Rich Blueprint Format (sections + inferenceType + instruction).

3. THE generated blueprint SHALL be saved as `status: "draft"` at `blueprints/<blueprint_id>/v1.json` in NOS and added to `_catalog.json` with `status: "draft"`.

4. THE response SHALL return the generated blueprint JSON and HTTP 201. THE blueprint SHALL NOT be loaded into the active BlueprintStore until explicitly activated via `PUT /blueprints/{id}` with `status: "active"`.

5. IF a blueprint with the given `blueprint_id` already exists, THE Endpoint SHALL return HTTP 409.

### Requirement 13: Logprob-Based Confidence Scoring

**User Story:** As a developer relying on confidence routing, I want per-field confidence derived from actual model token probabilities, so that routing decisions reflect genuine model uncertainty instead of a flat self-reported default.

#### Acceptance Criteria

1. THE Extractor SHALL request `logprobs: true` in every vLLM chat-completions call used for field extraction.

2. FOR each extracted field, THE Extractor SHALL compute Logprob_Confidence as `round(100 × exp(mean(logprob_i)))` over the tokens composing the field's value string in the generated output, clamped to [0, 100].

3. THE Extractor SHALL locate value tokens by reconstructing the generated text from the returned token sequence and mapping each field value's character span to its overlapping tokens. IF a value span cannot be located (e.g., value normalized differently than generated), THE Extractor SHALL fall back to the mean logprob of the full response, flagged via `confidence_source: "response_mean"`.

4. EACH FieldResult SHALL include `confidence_source`: `"logprobs"` | `"response_mean"` | `"model_reported"` | `"mock"`. The `"model_reported"` source is permitted only when logprobs are absent from the vLLM response.

5. `document_confidence` SHALL remain the mean of per-field confidences (Requirement 2.4) and the routing bands of Requirement 3 SHALL remain unchanged.

6. WHEN `mode="double_check"`, the verification pass SHALL also use Logprob_Confidence; the existing adjustment rules (confirmed +10, unconfirmed → 0/60) SHALL apply on top of logprob-based base scores.

7. WHEN `mode="auto"`, the classification confidence SHALL also be computed via Logprob_Confidence over the predicted `document_type` / `blueprint_id` tokens.

8. WHEN `MOCK_VLLM=1`, mock fixtures SHALL include synthetic logprobs so the logprob code path is exercised on CPU.

### Requirement 14: Guaranteed Schema Output via Guided Decoding

**User Story:** As a client of `/recognize`, I want the model output to be guaranteed valid against the blueprint schema, so that no field is ever lost to JSON parsing failures.

#### Acceptance Criteria

1. FOR `mode="blueprint"`, `mode="auto"` (extraction pass), and `mode="double_check"`, THE Extractor SHALL build a JSON Schema from the active Blueprint's normalized `fields[]` (object with one property per field, `type: ["string", "null"]`) and pass it to vLLM as `guided_json`.

2. WITH Guided_Decoding active, THE model response SHALL be parsed with a single `json.loads` — regex fallback SHALL NOT be needed. The fallback chain (Requirement 2.6) SHALL be retained only for deployments where the vLLM version lacks guided decoding support.

3. THE guided schema SHALL request flat scalar values (`{"field": "value"}`), NOT nested `{value, confidence}` objects — confidence comes from logprobs (Requirement 13), reducing output tokens and latency.

4. IF the vLLM guided-decoding backend returns an error for a given schema, THE Extractor SHALL retry once without `guided_json` and log `guided_json_fallback` — the request SHALL NOT fail solely because guided decoding is unavailable.

5. `POST /blueprints/generate` Pass 2 SHALL also use Guided_Decoding with a meta-schema of the Rich Blueprint Format, guaranteeing the generated blueprint is structurally valid.

### Requirement 15: Evaluation Job on MIDV-2020

**User Story:** As a contest judge or practitioner, I want a reproducible accuracy benchmark on a public dataset, so that I can verify the pipeline's real-world extraction quality myself.

#### Acceptance Criteria

1. THE repository SHALL include an Evaluation_Job (extension of `nebius-job` via `JOB_MODE=eval` or a dedicated entry point) that processes a labeled subset of MIDV-2020 and compares extraction results against ground truth.

2. THE repository SHALL include a setup script that downloads the MIDV-2020 subset from its public source, prepares images and a ground-truth manifest, and uploads both to NOS under `eval/midv2020/`.

3. THE evaluation manifest SHALL reference at least 50 documents covering at least 3 distinct document types (e.g., passports and ID cards from different countries).

4. FOR each document, THE Evaluation_Job SHALL record: per-field exact-match (after whitespace/case normalization), per-field normalized Levenshtein similarity, the field's Logprob_Confidence, and end-to-end latency.

5. THE Evaluation_Job SHALL write a summary report to NOS `eval/reports/<job_id>.json` containing: per-field accuracy, per-document-type accuracy, mean confidence for correct vs incorrect extractions (confidence calibration signal), latency percentiles (p50/p95), and total GPU cost estimate.

6. THE README and blog post SHALL present the evaluation results in a table, and the report generation SHALL be reproducible with a single documented command.

7. Blueprints needed for the evaluated MIDV-2020 document types SHALL be added as built-in blueprints (reusing `passport` where applicable; new ones, e.g., `id_card`, where not).

### Requirement 16: Document Packet Processing (Multi-Page Splitting)

**User Story:** As a client processing real-world uploads, I want to submit a multi-page PDF containing several documents and receive per-document extraction results, so that I don't have to split files manually.

#### Acceptance Criteria

1. THE Endpoint SHALL accept `mode="packet"` for PDF documents. WHEN `mode="packet"`, THE Endpoint SHALL convert every page (up to `PDF_MAX_PAGES`), classify each page independently (as in `mode="auto"`), group consecutive pages with the same classified `blueprint_id` into logical documents, and run extraction once per logical document.

2. THE `mode="packet"` response SHALL contain `documents: []`, each entry holding: `pages` (1-based page numbers), `blueprint_id`, `classification`, and the standard Recognition_Result fields (`fields`, `document_confidence`, `routing`).

3. THE packet response SHALL include a top-level `routing` equal to the most conservative routing across logical documents (escalate > review > auto).

4. WHEN a page classifies to no known blueprint, it SHALL form a logical document with `blueprint_id: null`, `routing: "escalate_to_operator"`, and `raw_text` extraction.

5. `mode="packet"` with a non-PDF document SHALL behave as `mode="auto"` (single image = packet of one).

6. THE per-request page limit SHALL be enforced: packets with more than `PDF_MAX_PAGES` pages SHALL return HTTP 422.

### Requirement 17: Demo UI

**User Story:** As a contest judge or evaluator, I want a zero-install browser demo, so that I can try recognition and see confidence visually within seconds.

#### Acceptance Criteria

1. THE Endpoint SHALL serve a single static demo page at `GET /demo` (no build step, vanilla HTML/JS served by FastAPI).

2. THE demo page SHALL allow: pasting the Bearer token, selecting a file (image or PDF), choosing mode and blueprint, and submitting to `/recognize` directly from the browser.

3. THE demo page SHALL render: extracted fields in a table with per-field confidence bars color-coded by routing band (green ≥85, yellow 50–84, red <50), `document_confidence`, `routing`, and latency.

4. WHEN bounding boxes are returned, THE demo page SHALL overlay them on the document image preview.

5. THE demo page SHALL include sample MIDV-2020 images selectable with one click (bundled or hot-linked), so a judge can run recognition without hunting for a test file.

6. THE demo page SHALL contain no hardcoded tokens or endpoints — endpoint URL defaults to `window.location.origin`, token always user-supplied.
